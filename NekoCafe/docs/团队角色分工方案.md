# NekoCafé 智慧餐饮预约平台团队角色分工方案

> 适用场景：5 人课程设计团队，按系统角色划分前后端开发任务。  
> 分工原则：每位成员负责一个角色相关的前后端功能；公共架构、登录认证、接口规范由组长统一维护；其他成员优先承担边界清晰、难度较低的 CRUD / 表格 / 按钮型功能。

---

## 1. 分工原则

本项目采用 **“按系统角色分工 + 组长负责公共架构与主流程”** 的方式。

不采用纯前端 / 纯后端分工，原因是：

1. 课程设计更需要每个人都有完整模块贡献；
2. 按角色分工更方便写个人职责和答辩说明；
3. 每个人负责自己角色下的前端页面和后端接口，边界更清晰；
4. 公共能力由组长统一维护，避免组员水平不一致导致架构混乱；
5. 简单角色优先分配给基础较弱的组员，降低开发风险。

---

## 2. 总体分工表

| 成员 | 负责角色 | 角色代码 | 难度 | 分工定位 |
|---|---|---|---|---|
| 组长 / 主力成员 | 顾客 + 公共架构 | `CUSTOMER` | 最高 | 负责主流程、登录认证、公共接口规范、最终集成 |
| 组员 A | 店员 | `STAFF` | 简单 | 负责预约签到、订单履约等按钮型功能 |
| 组员 B | 猫咪管家 | `CAT_CARETAKER` | 最简单 | 负责猫咪档案管理，主要是 CRUD |
| 组员 C | 店长 | `STORE_MANAGER` | 中等偏简单 | 负责门店信息、桌位、本店预约管理 |
| 组员 D | 管理员 / 总部运营 | `ADMIN` / `HQ_OPERATOR` | 中等 | 负责后台首页、数据看板、用户角色展示 |

## 3. 组长 / 主力成员：顾客 CUSTOMER + 公共架构

### 3.1 负责角色

- 顾客：`CUSTOMER`
- 公共能力：登录认证、JWT、安全拦截、接口规范、路由守卫、最终集成

### 3.2 负责原因

顾客端是整个项目最核心、最复杂的业务主线：

```text
注册登录 -> 浏览门店 -> 创建预约 -> 查看菜单 -> 创建订单 -> 支付 -> 查看个人记录
```

其他角色的功能大多依赖顾客端产生的数据，因此顾客主流程和公共架构由组长统一负责。

## 8. 难度分配建议

根据组员水平，可以这样安排：

| 能力水平 | 推荐角色 | 原因 |
|---|---|---|
| 最强 | 顾客 + 公共架构 | 需要处理登录、主流程、联调和集成 |
| 稍强 | 店长 | 涉及门店、桌位、预约，功能较多但不复杂 |
| 一般 | 管理员 / 总部运营 | 主要做统计卡片和列表展示 |
| 较弱 | 店员 | 主要做列表和按钮操作 |
| 最弱 | 猫咪管家 | 基础 CRUD，最容易完成 |

## 10. 前端开发模板要求

为了降低难度，除组长外，其他成员页面统一使用 Element Plus 基础组件。

允许使用：

```text
el-card
el-table
el-table-column
el-button
el-dialog
el-form
el-form-item
el-input
el-select
el-tag
el-message
```

页面结构统一为：

```text
1. 页面标题
2. 查询 / 新增按钮区域
3. el-table 数据表格
4. el-dialog 新增 / 编辑表单
```

不建议其他成员编写：

```text
复杂动画
复杂图表
复杂权限逻辑
复杂表单联动
复杂状态管理
```

## 11. 后端开发模板要求

每个角色模块尽量保持以下结构：

```text
controller/
service/
dto/
entity/
mapper/
```

Controller 只负责接收请求和返回结果：

```java
@GetMapping
@PostMapping
@PutMapping
@DeleteMapping
```

Service 只做基础逻辑：

```text
查询列表
查询详情
新增
修改
删除 / 改状态
```

不要让基础较弱的成员处理：

```text
复杂事务
复杂并发
复杂权限
复杂 SQL
复杂统计聚合
```

## 14. 协作规则

### 14.1 不互相修改页面

每个人只改自己角色对应的页面，不随意修改其他成员页面。

### 14.2 公共文件由组长维护

以下文件只由组长修改：

```text
frontend/src/router/index.ts
frontend/src/router/guards.ts
frontend/src/api/http.ts
frontend/src/stores/auth.ts
backend/src/main/java/com/nekocafe/security/
backend/src/main/resources/application.yml
backend/pom.xml
```

### 14.3 前端请求必须写在 `src/api/*.ts`

禁止在 Vue 页面里直接写 axios 请求。

推荐：

```text
src/api/staff.ts
src/api/cat.ts
src/api/manager.ts
src/api/admin.ts
src/api/dashboard.ts
```

### 14.4 后端返回格式统一

所有接口成功返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

所有接口失败返回：

```json
{
  "code": 400,
  "message": "请求参数不合法",
  "data": null
}
```

### 14.5 每个模块至少保留一个接口测试示例

例如：

```bash
curl http://localhost:8080/api/cats
curl http://localhost:8080/api/staff/reservations/today
curl http://localhost:8080/api/dashboard/summary
```

## 16. 总结

本项目最终采用以下分工：

```text
组长：CUSTOMER 顾客端 + 公共架构
组员 A：STAFF 店员端
组员 B：CAT_CARETAKER 猫咪管家端
组员 C：STORE_MANAGER 店长端
组员 D：ADMIN / HQ_OPERATOR 管理员与总部运营端
```

该方案既符合“每人负责一个角色前后端”的要求，又能把复杂的公共架构和核心主流程集中到组长手中，降低其他成员开发难度，提高项目按时完成的概率。
