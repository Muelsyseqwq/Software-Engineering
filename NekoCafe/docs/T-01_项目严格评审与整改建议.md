# NekoCafe 项目严格评审与整改建议

> 评审日期：2026-06-13  
> 评审对象：`选题指南/T-01_NekoCafé智慧餐饮预约平台.docx` 与当前 `NekoCafe` 项目实现  
> 评审视角：严格课程设计验收老师视角  
> 评审范围：前端、后端、工程质量、部署能力、测试与压测材料、答辩演示风险  
> 说明：本报告只整理项目不足与修改建议，不评价文档排版本身。

---

## 1. 评审结论摘要

当前 NekoCafe 项目已经具备较完整的业务形态：顾客、店员、店长、总部运营、猫咪管家等角色基本都有对应页面和接口；门店浏览、预约、排队、点单、沙箱支付、订单履约、数据看板、AI 推荐、猫咪健康档案、LBS 门店检索、活动优惠等功能均有实现痕迹。

但是，如果严格按照 T-01 选题指南验收，目前项目还存在几个明显短板：

1. `docker-compose.yml` 目前不能一键启动完整系统，只包含 Redis 和 MySQL，没有包含前端与后端服务。
2. 自动化测试严重不足，后端仅发现 2 个测试，前端未发现测试文件，距离“单元测试覆盖率 ≥ 60%，核心域 ≥ 80%”差距较大。
3. 未发现 k6 / JMeter 压测脚本与压测结果报告，无法证明“高峰期 1000 并发预约，核心接口 P95 ≤ 350 ms”。
4. 退款流程只有顾客申请与查询，缺少店长/总部审批、拒绝、打款确认等闭环。
5. 订单取消接口不明显，订单状态机还不够完整。
6. 店员侧部分后端操作疑似缺少门店归属校验，存在越权操作风险。
7. 会员等级、实名认证、异常告警等题目关键词实现不够突出。
8. 若演示数据不足，很多页面会出现空态，影响答辩观感。

总体判断：

> 项目不是“没做完”，而是“功能做得多，但工程验收闭环还没收紧”。如果只演示主流程，观感不错；如果老师按验收标准逐项检查，部署、测试、压测、退款闭环和权限边界会是主要扣分点。

## 5. 主要问题与整改建议

### P0：必须优先修复的问题

#### 5.1 `docker compose up` 不能一键启动完整系统

当前 `NekoCafe/docker-compose.yml` 主要包含：

- Redis
- MySQL local profile

缺少：

- 后端服务
- 前端服务
- 后端环境变量
- 前端访问后端配置
- 健康检查
- 数据库依赖启动顺序

这与选题指南“评委本地通过 `docker compose up` 一键启动”不一致。

**整改建议：**

补齐完整 compose：

1. `mysql`
2. `redis`
3. `backend`
4. `frontend` 或 `nginx + frontend dist`
5. `backend` 依赖 `mysql` 和 `redis`
6. 配置后端环境变量：
   - `MYSQL_HOST=mysql`
   - `MYSQL_PORT=3306`
   - `MYSQL_DATABASE=nekocafe`
   - `MYSQL_USERNAME=nekocafe_app`
   - `MYSQL_PASSWORD=change_me`
   - `JWT_SECRET=演示用强随机值`
7. 增加健康检查：
   - MySQL healthcheck
   - Backend `/actuator/health`
8. 在 README 或部署手册中写清楚：
   - `docker compose up --build`
   - 默认账号密码
   - 前端访问地址
   - 后端 API 地址

**扣分风险：高。** 评委如果执行 `docker compose up` 不能打开完整系统，会认为硬性验收不达标。

---

当前后端仅发现 2 个测试，前端未发现测试文件。虽然 `mvn test` 能通过，但不能证明覆盖率达标。

优先补核心业务测试，不要只补无关测试。

后端建议新增：

| `ReservationServiceTest` | 创建预约、重复预约冲突、取消预约、非本人取消失败 |

| `RecommendServiceTest` | 推荐排序、AI 不可用 fallback、用户偏好影响推荐结果 |

前端如果时间紧，至少补：

1. 登录页表单校验。
2. 预约创建页关键状态。
3. 订单页退款按钮显示逻辑。
4. 推荐页空态与有数据态。

**扣分风险：高。** 题目明确要求覆盖率，当前测试数量明显不足。

### P1：业务闭环不完整的问题

#### 5.4 订单取消接口不完整

当前 `OrderController` 主要看到：

- `POST /api/order`
- `GET /api/order/me`

证据：

- `backend/src/main/java/com/nekocafe/order/controller/OrderController.java:34`
- `backend/src/main/java/com/nekocafe/order/controller/OrderController.java:40`

未明显看到：

- 顾客取消订单。
- 店员/店长取消异常订单。
- 取消后库存、积分、优惠券回滚。

**整改建议：**

补充接口：

```http
POST /api/order/{id}/cancel
```

建议规则：

1. 只有订单本人可取消自己的未支付订单。
2. 已支付订单取消应进入退款流程。
3. 已制作或已完成订单不可直接取消。
4. 取消后释放优惠券或恢复可用状态。
5. 记录取消原因、取消时间。

---

#### 5.5 退款只有申请，没有审批/处理闭环

当前顾客端有退款申请与查询：

- `CustomerController.java:105`：`POST /orders/{orderId}/refunds`
- `CustomerController.java:114`：`GET /refunds/me`

但是没有看到店长/总部侧审批接口和前端退款管理页面。

**整改建议：**

后端补充：

```http
GET /api/manager/refunds
POST /api/manager/refunds/{id}/approve
POST /api/manager/refunds/{id}/reject
```

前端补充：

- 店长端“退款管理”tab。
- 展示退款申请、订单号、顾客、金额、原因、状态。
- 支持同意、拒绝、备注。

建议状态流转：

```text
REQUESTED -> APPROVED -> REFUNDED
REQUESTED -> REJECTED
```

如果时间紧，至少做“同意/拒绝”两个动作，支付打款可以标记为沙箱模拟。

---

#### 5.6 会员等级体系不完整

当前项目有积分、兑换、偏好，但会员等级更像字段存在，不像完整规则。

**整改建议：**

实现轻量等级规则：

| 等级 | 积分范围 |
|---|---:|
| 普通会员 | 0-999 |
| 银卡会员 | 1000-2999 |
| 金卡会员 | 3000-5999 |
| 黑金会员 | 6000+ |

并在：

1. 支付成功后自动更新等级。
2. 顾客个人中心展示等级、当前积分、距离下一等级还差多少。
3. 不同等级享受不同折扣或积分倍率。

---

#### 5.7 实名认证流程不明显

题目必做模块包含实名认证，但当前项目主线更偏注册登录和角色权限，没有明显实名认证流程。

**整改建议：**

课程项目无需接真实实名接口，可以做沙箱实名认证：

1. 顾客个人中心增加“实名认证”区域。
2. 填写姓名、身份证后四位或手机号。
3. 认证状态：`UNVERIFIED / PENDING / VERIFIED / REJECTED`。
4. 后台可以一键通过。
5. 文档说明“本项目使用沙箱实名认证”。

如果时间不够，至少做顾客端状态保存和演示数据。

---

#### 5.8 异常告警不明显

题目对店员后台明确要求“异常告警”。当前有状态处理，但告警概念不够突出。

**整改建议：**

做最小实现即可：

1. 桌位超时未清台告警。
2. 预约过号未处理告警。
3. 猫咪健康异常告警。
4. 订单长时间未出餐告警。

店员首页可增加“今日异常”模块，例如：

```text
今日异常
- 3 桌已超时 20 分钟未清台
- 2 个预约已过号
- 布偶猫「奶盖」今日健康状态异常
```

这样可以明确回应题目关键词。

### P1：权限与安全风险

#### 5.9 店员侧部分操作疑似缺少门店归属校验

后端 `StaffService` 中以下方法需要重点检查：

- `checkInReservation`：`backend/src/main/java/com/nekocafe/staff/service/StaffService.java:130`
- `startOrder`：`backend/src/main/java/com/nekocafe/staff/service/StaffService.java:157`
- `completeOrder`：`backend/src/main/java/com/nekocafe/staff/service/StaffService.java:171`
- `updateTableStatus`：`backend/src/main/java/com/nekocafe/staff/service/StaffService.java:239`

风险：如果只按订单 ID、预约 ID、桌位 ID 操作，而没有校验当前店员是否属于对应门店，一个 STAFF 账号理论上可能越权操作其他门店数据。

**整改建议：**

每个店员操作都应校验：

1. 当前用户是否为 STAFF。
2. 当前 STAFF 绑定了哪个门店。
3. 被操作的订单、预约、桌位、猫咪是否属于该门店。
4. 不属于则返回 403 或业务错误。

建议封装公共方法：

```java
private Long resolveStaffStoreId(Long staffId)
private void assertBelongsToStaffStore(Long staffId, Long storeId)
```

**注意：** 多角色系统不能只依靠前端隐藏菜单，后端也必须做权限边界校验。

### P2：演示体验和工程质量优化

#### 5.10 AI 推荐演示需要准备 fallback 方案

- `backend/src/main/java/com/nekocafe/recommend/ai/AiReasonProperties.java:10`
- `backend/src/main/java/com/nekocafe/recommend/ai/AiReasonProperties.java:13`

准备两种演示模式：

1. 无密钥稳定模式：规则推荐 + 模板推荐理由。
2. 有密钥增强模式：外部模型生成推荐文案。

注意：不要在代码注释、日志、报告、截图中暴露真实密钥或 token。

前端构建通过，但存在大 chunk 警告。

| 每家桌位 | 5-10 张 |

| 每只猫咪健康记录 | 3 条 |
| 顾客账号 | 1 个以上 |
| 店员账号 | 1 个以上 |
| 店长账号 | 1 个以上 |
| 总部账号 | 1 个以上 |
| 订单 | 至少 5 笔 |
| 活动 | 至少 3 条 |
| 排队票据 | 至少 2 条 |

订单建议覆盖：

- 待支付。
- 已支付。
- 制作中。
- 已完成。
- 退款中。

| D-03 概要设计说明书 | 系统架构、前后端分层、数据流、权限设计 |

| D-11 演示视频脚本 | 按角色走完整业务流程 |

## 9. 最终建议

接下来不要盲目继续加新功能，而应集中补齐验收闭环。最值得投入时间的 5 件事是：

1. 一键部署。
2. 测试覆盖。
3. 压测报告。
4. 退款/取消闭环。
5. 演示数据和演示脚本。

这 5 件补上后，项目会从“功能堆得多的课程 demo”变成“比较完整的软件工程课程设计交付”。
