# NekoCafé 角色权限与文件边界说明

> 适用范围：团队成员按系统角色分工开发前后端功能。  
> 核心原则：每个成员只修改自己角色相关页面、API 和后端包；登录、路由、权限、安全配置等公共文件由组长统一维护。

---

## 1. 角色与登录后首页

| 角色代码 | 角色名称 | 登录后默认首页 | 主要职责 |
|---|---|---|---|
| `CUSTOMER` | 顾客 | `/stores` | 浏览门店、预约、点单、支付、查看个人记录 |
| `STAFF` | 店员 | `/staff` | 今日预约、顾客签到、订单履约 |
| `CAT_CARETAKER` | 猫咪管家 | `/cats` | 猫咪档案、健康状态、互动信息维护 |
| `STORE_MANAGER` | 店长 | `/manager` | 门店信息、桌位、本店预约管理 |
| `HQ_OPERATOR` | 总部运营 | `/dashboard` | 平台经营数据和跨门店看板 |
| `ADMIN` | 系统管理员 | `/admin` | 用户、角色、门店和系统管理 |

多角色账号按以下优先级进入首页：

```text
ADMIN > HQ_OPERATOR > STORE_MANAGER > STAFF > CAT_CARETAKER > CUSTOMER
```

## 3. 前端公共文件：仅组长 / 集成人员维护

以下文件影响全部角色，普通组员不要自行修改：

```text
frontend/src/router/index.ts
frontend/src/router/guards.ts
frontend/src/router/permissions.ts
frontend/src/api/http.ts
frontend/src/stores/auth.ts
frontend/src/types/auth.ts
frontend/src/types/router.d.ts
frontend/src/layouts/AppLayout.vue
frontend/src/views/error/ForbiddenView.vue
```

用途说明：

| 文件 | 用途 |
|---|---|
| `frontend/src/router/index.ts` | 统一路由表和 `meta.roles` 配置 |
| `frontend/src/router/guards.ts` | 登录校验和角色访问校验 |
| `frontend/src/router/permissions.ts` | 角色首页、菜单配置、权限判断函数 |
| `frontend/src/api/http.ts` | token 注入、401/403 统一处理 |
| `frontend/src/stores/auth.ts` | 登录态、用户信息和角色列表 |
| `frontend/src/types/auth.ts` | 角色代码和认证数据类型 |
| `frontend/src/types/router.d.ts` | Vue Router meta 类型扩展 |
| `frontend/src/layouts/AppLayout.vue` | 根据角色过滤左侧菜单 |
| `frontend/src/views/error/ForbiddenView.vue` | 403 无权限页面 |

## 5. 后端公共文件：仅组长 / 集成人员维护

以下文件涉及全局安全、认证、异常处理和配置，普通组员不要自行修改：

```text
backend/src/main/java/com/nekocafe/auth/
backend/src/main/java/com/nekocafe/security/
backend/src/main/java/com/nekocafe/common/
backend/src/main/resources/application.yml
backend/src/main/resources/application-dev.yml
backend/pom.xml
```

重点说明：

- 不允许为了让页面能打开而删除 `@PreAuthorize`。
- 不允许把 `SecurityConfig` 中的 `.anyRequest().authenticated()` 改成全部放行。
- 不允许在日志、注释或测试输出中打印 token、密钥或完整认证头。

## 7. 新增路由或接口时的协作规则

1. 组员需要新增页面时，只在自己的 `views/<role>/` 目录中创建页面。
2. 新页面需要加入菜单或路由时，把路径、页面名、角色代码发给组长，由组长统一修改 `router/index.ts` 和 `permissions.ts`。
3. 组员需要新增接口时，只在自己的后端包中新增 Controller / Service / DTO。
4. 新接口必须声明角色权限，不能新增“任意登录用户都能访问”的角色业务接口。
5. 如果某个接口确实需要多个角色访问，先说明原因，再由组长统一确认权限表达式。
6. 前端页面不要直接写 axios 请求，应统一调用 `frontend/src/api/*.ts`。
7. 页面出现 403 时，不要删除权限控制，应检查当前账号角色是否正确。

## 9. 答辩说明建议

可以在答辩中强调：

- 项目采用“按系统角色分工”，每个成员有独立前后端模块。
- 前端不是简单隐藏菜单，而是使用 `meta.roles + 路由守卫 + 403 页面` 控制页面访问。
- 后端使用 Spring Security 方法级鉴权，保证直接调用接口也不能越权。
- 公共安全文件由组长维护，减少多人修改造成的权限漏洞和合并冲突。
