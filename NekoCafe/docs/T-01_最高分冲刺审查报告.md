## 1. 总体结论

当前项目已经具备较完整的功能雏形：前后端分层清晰，角色页面较多，数据库迁移较丰富，预约、点单、支付、后台管理、数据看板、猫咪档案等模块均有一定实现。

但如果目标是“最高分”，当前项目仍存在明显短板。核心问题不是“没有功能”，而是缺少最高分所要求的工程证据链：

1. 没有真正满足 `docker compose up` 一键启动完整系统。
2. 自动化测试和覆盖率证据基本缺失。
3. 没有 k6 / JMeter 压测脚本和结果。
4. 12 份产出文件 + 5 份管理表单没有形成完整交付包。
5. 安全配置仍有弱默认值。
6. 核心业务存在并发、库存扣减、状态机完整性风险。
7. T-01 特色功能“AI 推荐 / 猫咪健康趋势”还不够突出。
8. 文档偏说明型，尚未形成可评分、可追溯、可审计的证据型文档。

严厉评价：

> 当前项目可以作为“能演示的课程设计项目”，但还不像“最高分交付包”。  
> 冲最高分的关键，不是继续盲目增加页面，而是补齐一键部署、测试覆盖、性能压测、安全自查、交付文档和演示闭环。

### 5.1 安全配置存在弱默认值

#### 问题描述

当前配置中存在弱默认值：

```yaml
username: ${MYSQL_USERNAME:zhaoyh}
password: ${MYSQL_PASSWORD:zhaoyh666}
```

以及：

```yaml
secret: ${JWT_SECRET:change_me_to_at_least_32_chars}
```

问题在于：

- 默认数据库账号密码不应出现在配置中。

- 容易被评委认为 Secret 管理不规范。

当前配置会影响安全合规得分。

将配置改为强制环境变量：

```yaml
username: ${MYSQL_USERNAME}
```

- 鉴权。

- XSS。

- 文件上传。
- 密码存储。
- 敏感信息。

### 5.2 订单创建没有真正扣减菜品库存

#### 问题描述

当前 `OrderService.create` 会检查菜品库存：

```java
if (dish.getStock() != null && dish.getStock() < item.quantity()) {
    throw new BizException(3005, dish.getName() + "库存不足");
}
```

但创建订单后没有真正扣减菜品库存。

这意味着：

- 库存检查不能防止超卖。
- 多人同时点同一道菜时，库存数据不变。
- 高并发下业务数据会失真。

#### 扣分风险

这会影响“数据完整、并发安全”评分。评委如果追问并发点单或库存一致性，很容易被问穿。

#### 建议改进

订单创建时增加条件扣减：

```sql
UPDATE dish
SET stock = stock - ?
WHERE id = ?
  AND stock >= ?
```

Java 逻辑中：

1. 校验菜品存在、上架、属于门店。
2. 对每个菜品执行条件扣减。
3. 如果任一菜品扣减失败，抛异常并回滚事务。
4. 创建订单和订单项。

必须配套测试：

- 库存足够，下单成功。
- 库存不足，下单失败。
- 并发下单不会超卖。

---

### 5.3 订单状态机和退款闭环不够强

#### 问题描述

当前项目已经有订单、支付、退款申请等功能，但最高分要求的是完整业务闭环：

```text
下单 → 支付 → 接单 → 制作 → 完成 → 评价 / 退款
```

目前还需要进一步证明：

- 订单状态有哪些。
- 各状态允许哪些角色触发。
- 非法状态跳转如何拦截。
- 退款如何从申请到审核再到完成 / 拒绝。
- 订单、支付、退款之间的数据一致性如何保证。

#### 扣分风险

T-01 M4 明确要求“订单状态机 / 取消改约 / 退款”。如果状态机没有文档、没有测试、没有完整闭环，会影响功能完成度和详细设计评分。

#### 建议改进

在 D-04 详细设计中补订单状态机图：

```text
CREATED
  -> PAID
  -> PREPARING
  -> COMPLETED
  -> REFUNDING
  -> REFUNDED / REFUND_REJECTED
  -> CANCELLED
```

同时补状态转移表：

| 当前状态 | 操作 | 目标状态 | 角色 | 约束 |
|---|---|---|---|---|
| CREATED | 支付 | PAID | 顾客 | 订单归属本人 |
| PAID | 接单 | PREPARING | 店员 | 所属门店 |
| PREPARING | 完成 | COMPLETED | 店员 | 已接单 |
| PAID / PREPARING | 申请退款 | REFUNDING | 顾客 | 未申请过退款 |
| REFUNDING | 同意退款 | REFUNDED | 店长 / 总部 | 审核通过 |
| REFUNDING | 拒绝退款 | REFUND_REJECTED | 店长 / 总部 | 填写原因 |

配套增加测试，证明非法跳转会被拒绝。

### 5.4 AI 推荐特色不够明显

#### 问题描述

T-01 的业务目标中明确提到：

> 系统须支持 AI 智能推荐，根据用户偏好与同行猫咪性格匹配桌位与菜品。

当前项目已有用户偏好相关能力，但尚未形成完整推荐闭环。缺少明显的：

- 推荐接口。
- 推荐算法服务。
- 推荐理由。
- 推荐桌位。
- 推荐菜品。
- 匹配猫咪性格。
- 前端推荐展示区域。

如果只是保存偏好标签，不足以支撑“AI 推荐”或“智能推荐”这一亮点。

#### 扣分风险

T-01 的项目特色就是猫咖与智能推荐。如果没有推荐展示，项目会变成普通餐饮预约系统，特色不足。

#### 建议改进

时间紧时可以采用“可解释规则推荐”，作为 AI 推荐的降级实现。

新增接口：

```text
GET /api/recommendations
```

返回示例：

```json
{
  "recommendedTables": [],
  "recommendedDishes": [],
  "matchedCats": [],
  "reason": "你偏好安静区，推荐靠窗 2 人桌；同行猫咪性格温顺，适合新手互动。"
}
```

前端展示：

- 今日推荐桌位。
- 适合你的猫咪。
- 推荐菜品。
- 推荐理由。

文档说明：

> 本项目采用基于用户偏好、猫咪性格、桌位区域和菜品标签的可解释规则推荐，作为 AI 推荐功能的课程设计实现版本。

```text
cat_health_record
```

这会显著增强猫咖主题和选做功能完成度。

### 5.6 CI/CD 仍停留在基础构建层面

#### 问题描述

当前 CI 已经能执行基础构建，但最高分角度还缺少：

- 覆盖率报告。
- 覆盖率阈值。
- 前端真正 lint。
- 后端 package artifact。
- Docker 镜像构建。
- 安全扫描。
- E2E 测试。

前端当前 `lint` 实际只是 typecheck：

```json
"lint": "vue-tsc -b --noEmit"
```

这不是真正的 ESLint 检查。

#### 建议改进

CI 建议至少增加：

1. `backend-test-with-coverage`
2. `frontend-lint`
3. `frontend-test`
4. `docker-build`
5. 覆盖率报告上传 artifact

如果时间不足，最低限度要让 CI 产出 JaCoCo 覆盖率报告。

## 6. P2 级问题：影响演示体验和答辩表现

### 6.1 登录页演示账号体验不够顺滑

当前 README 中写了统一演示密码，但登录页角色卡只展示账号。如果评委或组员演示时还要翻 README 找密码，会降低演示流畅度。

建议：

1. 点击角色卡时自动填充账号和统一演示密码。
2. 或在登录页明显展示：

```text
统一演示密码：NekoCafe@2026
```

这是很小的改动，但能显著降低答辩翻车概率。

---

### 6.2 部分前端错误被当成空数据

一些页面在接口失败时容易显示“暂无数据”，而不是明确的错误状态。这样在演示中会造成误判：后端接口失败，页面却像是没有数据。

建议关键页面统一具备：

- loading 状态。
- error 状态。
- empty 状态。
- retry 按钮。

不要把 error 当 empty。

优先处理：

- 订单页。
- 结算页。
- 店长管理页。
- 总部活动发布页。
- 数据看板页。

---

### 6.3 移动端适配需要强化

T-01 提到用户可通过移动端 / 小程序 / Web 完成操作。如果没有小程序，Web 至少要具备较好的 H5 体验。

当前后台侧边栏和表格页面在窄屏下可能不够友好。

建议：

- 手机宽度下侧边栏折叠。
- 顾客端页面优先卡片化。
- 表格在移动端改成卡片列表。
- 关键操作按钮固定在底部或显著位置。

---

### 6.4 数据看板要补公式说明

T-01 要求数据看板展示：

- 坪效。
- 翻台率。
- 会员复购率。

当前项目已有相关计算，但文档和答辩中必须解释公式，否则评委会认为只是“堆图表”。

建议在 D-03 / D-04 / D-06 / PPT 中写明：

```text
坪效 = 周期内门店营收 / 门店面积
翻台率 = 已签到或完成预约数 / 桌位数 / 天数
复购率 = 周期内多次消费会员数 / 消费会员总数
```

同时说明数据来源表：

- `payment_record`
- `food_order`
- `reservation`
- `store`
- `dining_table`
- `member_account`

## 12. 最终评价

当前 NekoCafe 项目已经具备冲高分的基础：功能模块多、角色覆盖广、前后端技术栈合理、数据库迁移较完整、预约并发控制有亮点。

但要拿最高分，必须从“功能实现”升级为“工程化交付”：

```text
功能完整
+ 一键运行
+ 测试覆盖
+ 性能数据
+ 安全自查
+ 文档齐全
+ 演示稳定
+ 答辩能解释
```

接下来最关键的四件事是：

1. 补完整 Docker Compose 一键启动。
2. 补测试覆盖率和测试报告。
3. 补 k6 / JMeter 压测脚本与结果。
4. 补齐 12 + 5 交付物和索引矩阵。

严厉地说：

> 现在项目“能做出来”的部分已经不少，但“能证明它高质量”的证据还不够。  
> 最高分不是靠页面数量堆出来的，而是靠工程闭环、质量数据、文档追溯和稳定演示拿到的。
